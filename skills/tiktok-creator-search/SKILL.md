# TikTok Creator Search — 达人搜索

在 TikTok Shop Creator Marketplace 中按多维度筛选搜索达人，支持在单轮搜索的 60 人上限内继续翻页。

本文件是达人搜索专项行为的权威来源；筛选字段、自然语言映射、范围和互斥关系以 [references/creator-search-filters.md](references/creator-search-filters.md) 为准。Agent 只保留搜索能力摘要与跨能力编排。

## 触发条件

当用户提到以下意图时加载本技能：
- "搜索/找/筛选 TikTok 达人/creator"
- "帮我找美妆/3C/服装类目的达人"
- "查找 GMV 在 xxx 区间的达人"
- "搜索粉丝画像为 xxx 的达人"
- "可以按哪些条件筛选达人"
- 任何包含达人搜索、市场筛选的请求

## 执行方式（MCP 优先）

### 类目筛选前置步骤（重要）

如果用户提到了品类/类目关键词（如"美妆"、"3C"、"beauty"、"electronics"），**必须**先从当前授权店铺的 `get_categories` 结果获取类目 ID，再调用 `search_creators`：

1. **店铺市场类目树** — 对当前支持的 SEA 店铺调用 `get_categories` 时传 `category_version: "v2"`，不传 `locale`；TikTok 按当前 `shop_cipher` 使用店铺市场的默认语言和类目 ID。
   - 用户表达宽泛品类时，首次不传 `keyword`，获取当前店铺的顶层类目，再按当地语言 `local_name` 与用户意图进行语义匹配。
   - 只有用户明确指定具体子类目时，才将该意图翻译为店铺当地语言关键词后传入 `keyword`；不得将中文关键词与其他市场 locale 组合获取过滤 ID。翻译或匹配不唯一时使用 WorkBuddy 选项交互，不猜测 ID。
   - **顶层定义**：只有 `parent_id == "0"` 的类目才是达人搜索默认使用的顶层类目；`is_leaf=false` 只能说明不是叶子节点，不能据此把中间层当作顶层。
   - **关键层级约束（与版本无关）**：达人搜索的 `category` 过滤只支持顶层类目及其直接下一级子类目（TikTok 官方约束："A child category is only 1 level deeper than the parent"）。禁止把更深层命中项直接作为搜索子类目，否则过滤可能被静默忽略。
2. **默认匹配顶层类目** — 用户只表达宽泛品类或未明确点名子类目时，只使用 `parent_id == "0"` 的顶层类目：
   - 构造 `category` 参数时只传 `parent_category_id`，不传 `child_category_id_list`：`[{parent_category_id: "顶层ID"}]`
   - 关键词结果包含对应顶层类目时直接使用；只返回中间层或深层命中项时，复用同一店铺不带 `keyword`、不带 `locale`、`category_version: "v2"` 的顶层列表，按命中项的类目归属选择对应顶层
   - 无法从工具结果可靠确认深层命中项所属顶层时，不猜测类目 ID；按下方“WorkBuddy 选项交互”让用户从相关顶层类目中选择
   - 匹配到多个同样合理、无法从用户原话判定的顶层类目时，调用 `AskUserQuestion` 多选；唯一明确匹配时直接继续
3. **仅按用户明确意图使用子类目** — 只有用户原话明确点名具体子类目时，才允许传 `child_category_id_list`：
   - 子类目必须是所传顶层类目的直接下一级，即子类目的 `parent_id` 等于该顶层类目的 `id`
   - 构造：`[{parent_category_id: "顶层ID", child_category_id_list: ["直接子类ID1"]}]`
   - 用户明确点名的是更深层类目时，将其映射到所属的直接子类目；无法从工具结果可靠完成映射时，回退为只传顶层类目，不猜测或传入深层 ID
   - 仅因关键词查询返回子类目，或没有合适的顶层名称匹配，不视为用户明确指定子类目，不得自动降级使用子类目
4. 将构造好的 `category` 参数传给 `search_creators`

> **核心原则**：默认顶层覆盖；只有用户明确指定子类目时才缩小范围。层级只能到“顶层类目 + 直接子类目”，宁愿回退顶层多搜，也不要猜测或使用深层 ID。

调用 MCP 工具 **`search_creators`**，不要写脚本。若已授权可直接调用；未授权时先走 `authorize`。

授权、限流、MCP 不可用和本地运行环境恢复遵循 Agent 的权威规则，本技能不重复展开。

### WorkBuddy 选项交互

- 只在零条件搜索、类目候选存在真实歧义、用户主动要求选择筛选档位、筛选表达无法唯一映射或出现互斥字段时调用 `AskUserQuestion`。用户已给出的可执行条件不二次确认。
- 类目与 GMV 问题均设置 `multiSelect: true`，每个问题提供 2–4 个选项。类目标签使用本地化名称，描述展示父级路径或层级，内部映射原始类目 ID；不展示 ID，也不从显示文本反推 ID。
- 类目候选超过 4 个时，按规范化名称完全匹配、名称包含关键词、工具原始返回顺序排序，只展示前 4 个。类目自定义输入作为新 `keyword` 重新调用 `get_categories`；唯一匹配时继续，仍歧义时再次发起选项交互，不得把自由文本当作类目 ID。
- GMV、销量、年龄和其他档位选项与参数映射遵循筛选字段 Reference。自定义输入不能精确映射时提示用户改选，不静默放宽。
- 两个条件同时需要确认时，在同一次 `AskUserQuestion` 中提交两个问题。取消或空选择时不调用 `search_creators`；工具不可用或失败时，降级为最多 4 个带编号的文本选项，不使用 Markdown 表格。

### 画像发现与零条件搜索

- 用户询问支持哪些条件时，不调用搜索，按“品类与身份 / 商业表现 / 粉丝受众 / 内容与合作特征”四组给出用户可读摘要和一个完整示例；不展示内部字段或枚举。
- 用户只表达“帮我找达人”等搜索意图，且没有提供任何可执行条件时，不立即调用 `search_creators`。先用 `AskUserQuestion` 提供上述四个画像维度，设置 `multiSelect: true`；允许用户通过自定义输入直接描述完整画像或输入“直接搜索”。选择维度后只追问所选维度的具体值。
- 用户明确回复“直接搜索”时按默认 20 位执行零条件搜索。只要用户已提供至少一个可执行条件，就直接搜索；除真实歧义、非法范围或互斥条件外，不强制补全其他画像维度。

### 入参

| 参数 | 说明 |
|------|------|
| `keyword` | 用户名/昵称搜索（可选） |
| `category` | 品类筛选数组（可选）。默认使用顶层模式：`[{parent_category_id: "顶层ID"}]`，其中顶层类目的 `parent_id == "0"`。只有用户明确指定具体子类目时才使用顶层+直接子类目模式：`[{parent_category_id: "顶层ID", child_category_id_list: ["直接子类ID1"]}]`；直接子类目的 `parent_id` 必须等于顶层 ID。多个对象 OR 关系，单个对象内 child_category_id_list AND 关系。类目 ID 必须先通过 `get_categories` 获取，不得猜测或使用深层 ID。 |
| `gmv_ranges` / `units_sold_ranges` | GMV 与销量档位（可选） |
| `follower_demographics` | 粉丝数、年龄段、主要性别及最低占比（可选） |
| `content_performance` | 视频、带货视频、互动率和直播观看表现（可选） |
| `affiliate_data` | 佣金、样品后发帖表现、达人归属、快速增长与近 90 天未邀约（可选） |
| `page_size` | 每页条数，默认/上限 20 |
| `page_token` / `search_key` | 首次不传；后续翻页同时复用上一页的 `next_page_token` 和首次响应的 `search_key` |

所有字段映射和校验规则只在筛选字段 Reference 中维护，本 Skill 不复制实现级枚举。

### 搜索会话与人数上限

- 用户的已明确条件构成达人画像；不新增等待用户二次确认的回合。类目解析、类目过滤闭环校验和必要的回退重搜完成后，才以最终实际生效的条件作为本轮搜索条件。
- 将同一组规范化搜索条件下的首次搜索和“继续翻页”视为同一搜索会话。用户修改任一搜索条件或明确要求重新搜索时，创建新的搜索会话并重置人数预算。
- 用户未指定数量时，默认展示 20 位达人。用户指定正整数时，目标展示数为该数量；超过 60 时按 60 执行，并在结果列表前明确说明“本次搜索最多展示 60 位达人”。数量小于 1 或无法识别时，要求用户给出 1–60 的数量。
- 每次工具调用的 `page_size` 保持 20。为满足目标数量可顺序使用内部 `page_token` 拉取后续页，并在每个后续请求中同时传入首次响应的 `search_key`；展示到目标数量、无下一页或累计展示 60 位时必须停止。最后一页只展示仍在目标范围内的记录。
- 必须在当前对话中保留搜索条件、已展示数量、下一页令牌和 `search_key`。用户继续翻页时，最多再展示 `min(20, 60 - 已展示数量)` 位；内部令牌与搜索键永不对外展示。

### 输出与后续（固定格式）

禁止直接展示 MCP 原始 JSON、内部字段、枚举、类目 ID、分页令牌、搜索键或目标展示人数。每次成功搜索都按以下顺序输出：

1. **结果摘要**：实际展示数、`total_count`、当前排序、是否还有下一页，以及一行用户可读的“已应用画像”。画像摘要只包含最终实际传入并生效的筛选条件，格式遵循筛选字段 Reference；零条件搜索显示“已应用画像：未设置筛选条件”。
2. **达人列表**：固定展示“排名 / 昵称 / 用户名 / 粉丝数 / 近 30 天 GMV 区间 / 类目 / 达人 ID”。
3. **分页状态**：有更多结果且未达到 60 人时，使用“已显示 X / 共 Y；本轮最多展示 60 位达人，还可继续获取最多 Z 位”。没有下一页时使用“已显示全部结果”。达到 60 人且仍有更多结果时，固定说明“本次搜索已达到 60 位达人上限；请调整条件后重新搜索”，不再建议继续翻页。不要向用户展示 `page_token`。
4. **下一步**：建议分析选中达人、在剩余预算内继续翻页或调整条件；新增建议“选择 2 位及以上达人进行相对评分排名”。必须等待用户明确指定候选人后才调用 `creator_performance`，不要擅自开始批量分析或建联。

格式化规则：

- 默认保持 `search_creators` 返回顺序并以此编号；只有用户明确要求按某字段排序时才调整，并在摘要说明排序依据。
- 若 `username` 可用，达人主页链接统一拼接为 `https://www.tiktok.com/@{username}`。主表中“昵称”列应链接到该主页；“用户名”列继续显示纯文本 `@username`，不要改成第二个视觉重复的链接。
- 粉丝数使用千位分隔；GMV 区间转换为用户可读值（`0–100` / `100–1K` / `1K–10K` / `10K+`）；缺失字段统一显示 `—`，不要猜测。
- `username` 缺失、为空或明显不合法时，昵称回退为纯文本，不要猜测用户名，也不要用 `nickname`、`creator_user_id`、`profile_tt_uri` 或其他字段拼接链接。
- “类目”列只读取 `category_details[].local_name`，不得展示 `category_details[].id` 或 `category_ids`。按上游 ID 顺序展示并去重：主列表最多显示 3 个名称，更多显示“+N”；HTML 展开详情展示全部名称。
- `category_details[].local_name` 使用当前店铺的当地语言，不翻译为对话语言；类目 ID 和名称必须来自同一个店铺市场上下文。
- 没有 `category_ids` 时类目显示 `—`。存在 ID 但没有名称的条目合并为“未识别类目（N）”；类目解析整体失败时也按此规则降级，不影响达人列表展示。
- 类目过滤闭环校验继续使用原始 `category_ids` 与筛选 ID 比较；不得使用本地化名称判断是否命中，过滤疑似失效时不要展示为合格结果。
- 必须为每位达人保留搜索结果返回的 `creator_open_id`。调用 `creator_performance` 时将该值放入兼容入参 `creator_user_id`；调用 `create_conversation` 时将同一个值放入 `creator_open_id`。不得截断、改写或从其他字段推断。
- 将每位达人的 `selection_region` 与当前授权店铺市场代码比较。不匹配的达人不得作为合格结果展示，也不得进入分析或建联；全部结果均不匹配时，停止本轮并提示检查当前授权店铺后重新搜索。`selection_region` 缺失时保留结果但标记“地区未校验”，不猜测。
- Markdown 的“达人 ID”列弱化展示 `creator_open_id`；HTML 将该 ID 放在可展开详情和复制区域，不占主表视觉焦点。
- `creator_open_id` 缺失时显示“达人 ID 不可用”，并明确该达人暂不可分析或建联。
- 结果为空时不生成空表。固定输出“未找到符合条件的达人”，并建议放宽类目、GMV 或关键词，不复述搜索条件。
- 单页最多 20 条；同一搜索会话最多展示 60 位达人，不承诺获取全量结果。

### WorkBuddy HTML 可视化

- HTML 报告的共享品牌配色遵循 Agent 的“WorkBuddy HTML 共享配色”契约，本 Skill 不自行定义另一套主色或色阶。
- 结果为 1–5 条时使用 Markdown 表格。在 **WorkBuddy** 中达到 **6 条**时，必须通过当前宿主的 HTML 产物通道创建自包含的响应式 HTML 报告；用户明确要求 HTML 时不受条数限制。非 WorkBuddy 宿主使用 Markdown。
- 在 WorkBuddy 中，不得仅因宿主未显式声明 HTML/可视化能力而回退 Markdown。仅当 HTML 产物实际创建失败时，必须回退为包含全部记录的 Markdown，不能只给摘要或丢失达人，并如实说明未生成报告。
- HTML 搜索报告包含用户可读的“已应用画像”，但不得包含内部字段、枚举、类目 ID、搜索键、分页令牌或目标展示人数；主表类目最多显示 3 个名称和“+N”，展开详情显示全部本地化名称及合并后的“未识别类目（N）”。其余包含返回数/总数/分页状态、可搜索和排序的达人表格、达人详情、达人 ID 复制区域和下一步建议；下一步建议包含“选择 2 位及以上达人进行相对评分排名”。不添加无法调用 Agent 或 MCP 的伪按钮，也不自动开始分析；达到 60 人上限时必须展示上限提示而非继续翻页入口。
- HTML 报告中的昵称链接使用 `<a>` 指向 `https://www.tiktok.com/@{username}`，并带 `target="_blank"` 与 `rel="noopener noreferrer"`；链接文本与 URL 都必须先安全转义/编码。
- 只使用内联 CSS/JavaScript，不依赖 CDN、远程字体或外部图片；只实现真实可用的搜索、排序、展开和复制交互，不添加无法触发 Agent 或 MCP 操作的伪按钮。
- 昵称、用户名及所有工具返回文本写入 HTML 前必须转义；禁止把工具内容作为 HTML 或脚本执行。
- Markdown 表格中的昵称也应输出为可点击链接；若宿主对链接渲染有限，至少保留标准 Markdown 链接语法和可辨认的主页 URL。
- 生成 HTML 后仍需在对话中按“结果数量/排序/分页/已应用画像”的顺序给出简短摘要。只有实际生成产物时才可声明存在 HTML，不得虚构文件名或 `outputs` 路径。

### 类目过滤闭环校验（防静默失效）

用了 `category` 过滤时，搜索返回后**必须自检**：把返回结果里各达人的 `category_ids` 汇总，与本次传入的 `parent_category_id` / `child_category_id_list` 比对。

- 若**零重叠**（返回的 `category_ids` 完全不含所传的类目 ID），说明类目 ID 很可能层级过深或版本不对，**过滤已静默失效**——不要把这批结果当作"符合品类"直接给用户。此时应：
  1. 提示用户"当前类目过滤可能未生效"；
  2. 回退到更浅的类目层级重搜：把所传的深层子类目换成其**父类目**（只传 `parent_category_id`）重新 `search_creators`；仍零重叠时再上溯一级到顶级父类目。
- 若有重叠，说明过滤生效，正常展示。

## 市场限制

当前仅支持以下东南亚市场：

| 国家 | 代码 |
|------|------|
| 泰国 | TH |
| 马来西亚 | MY |
| 越南 | VN |
| 菲律宾 | PH |
| 印度尼西亚 | ID |
| 新加坡 | SG |

用户在搜索前若明确指定目标市场（如"帮我找日本的达人"、"搜索美国市场创作者"），应**立即告知不支持该市场**，并建议用户改选以上 SEA 市场之一，不要直接调用 `search_creators`。

## 错误处理

搜索专属错误分支仅处理市场不支持、类目参数无效、搜索条件不合法和类目闭环校验失败；其余授权、限流、MCP 可用性与本地环境问题均遵循 Agent 权威规则。
